最近看到很多人在討論 Loop Engineering . 如果回頭看過去一年 AI Coding 的發展，我反而覺得大家其實是在一路往上抽象。 一開始我們研究怎麼寫 Prompt，後來發現光會下指令不夠，又開始想辦法管理 Context，避免模型每次都像剛報到的新同事。接著進入 Harness 階段，讓 Agent 不只是聊天，而是真的能讀檔、改檔、跑測試、開 PR。 但有趣的是，當 Agent 終於能做事之後，真正麻煩的問題才出現。 例如工作從哪裡來？失敗了怎麼辦？誰來驗證結果？什麼時候該停下來？又什麼時候該把人找回來？ . 這些問題其實都不是 Prompt 或 Harness 能解決的。 這也是我理解的 Loop Engineering。 它不是把 Prompt 寫得更長，也不是讓 Agent 24 小時不眠不休地一直跑。剛好相反，它比較像是在幫 Agent 建立一套交通規則：知道自己現在在哪裡、剛剛做了什麼、接下來該往哪裡走，以及什麼情況下必須停車。 . 最近我越來越有一個感受： Loop 很重要，但 Loop 本身不是資產。 真正的資產，是那些被沉澱下來的經驗與規則。 如果每次都要重新教 Agent 專案架構、Coding Style、測試方法和 Review 標準，那不管 Loop 多漂亮，本質上都只是成本比較高的 while(true)。 真正有價值的是讓這些知識一次次累積下來，讓 Agent 不用每輪都從實習生模式重新開始。 所以真正的分水嶺會是： 開始思考「我要設計什麼系統，讓 AI 持續做對的事情」。 我覺得這就是 Loop Engineering 真正有意思的地方。
